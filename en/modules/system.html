System Module

Tagtag Starter system module documentation, including system configuration, dictionary management, and log management.

The System module is one of the foundational modules of the Tagtag Starter framework, providing basic functionality support for system configuration, data dictionaries, operation logs, and more.

1. Dictionary Management (Dict)

Dictionary management is used to dynamically manage system-wide constants and enumeration values, reducing frontend hardcoding and improving system flexibility and maintainability.

1.1 Data Model

1.1.1 Dictionary Type (DictType)

Attributes:

  • Type code (type_code): Unique identifier, such as sys_user_sex
  • Type name (type_name): Display name, such as "User Gender"
  • Status (status): Enabled/Disabled
  • Description (description): Type description
  • Sort order (sort): Display order
  • Create time (create_time): Creation time
  • Update time (update_time): Update time

1.1.2 Dictionary Data (DictData)

Attributes:

  • Dictionary type code (type_code): Associated dictionary type
  • Data value (value): Dictionary value, such as 1
  • Label (label): Display label, such as "Male"
  • Status (status): Enabled/Disabled
  • Sort order (sort): Display order
  • Description (description): Data description
  • Create time (create_time): Creation time
  • Update time (update_time): Update time

1.2 Core Functionality

Functions:

  • CRUD operations for dictionary types
  • CRUD operations for dictionary data
  • Dictionary cache management
  • Dictionary import/export

Implementation Details:

1.2.1 Dictionary Caching Mechanism

  • Uses Redis to cache dictionary data, improving read performance
  • Supports automatic cache refresh
  • Automatically clears cache when dictionary is updated
  • Cache key format: dict:{type_code}

Cache Configuration:

# Dictionary cache configuration
dict:
  cache:
    enabled: true
    expire: 3600 # Cache expiration time (seconds)

1.2.2 Dictionary Usage

Frontend Best Practice:

Use the useDict composable function to elegantly manage dictionary data and reactive state.

<script setup lang="ts">
// Elegant destructuring, supports getting multiple dictionaries at once
const { sys_user_sex, sys_notice_type } = useDict('sys_user_sex', 'sys_notice_type');
</script>

<template>
  <!-- Direct binding to options -->
  <Select v-model="form.sex" :options="sys_user_sex" />
</template>

Backend Best Practice:

The backend retrieves dictionary data through interfaces provided by the Contract Layer, achieving decoupling between modules.

  1. Define Contract Interface: Define the DictApi interface in tagtag-contract-system.
public interface DictApi {
    /**
     * Get data list by dictionary type
     */
    List<DictItemDTO> getDictData(String typeCode);

    /**
     * Get dictionary label (translation)
     */
    String getDictLabel(String typeCode, String itemValue);
}
  1. Inject and Use: Inject DictApi in other modules (such as User module) and call the interface directly to get values.
@Service
@RequiredArgsConstructor
public class UserServiceImpl implements UserService {

    private final DictApi dictApi;

    public UserVO getUser(Long id) {
        User user = userMapper.selectById(id);
        UserVO vo = toVO(user);
        
        // Use contract interface to get dictionary label (value retrieval only)
        vo.setSexLabel(dictApi.getDictLabel("sys_user_sex", user.getSex()));
        
        return vo;
    }
}

1.3 Best Practices

  • Naming Convention: Dictionary type codes use the {module}_{function}_{field} format, such as sys_user_sex
  • Cache Management: Regularly clean up expired dictionary cache
  • Data Maintenance: Regularly review dictionary data and clean up unused data in a timely manner
  • Permission Control: Strictly control dictionary management permissions to avoid accidental operations

2. Message Management (Message)

Message management is used for in-system message notifications, supporting multiple message types and sending methods.

2.1 Data Model

2.1.1 Message Template (MessageTemplate)

Attributes:

  • Template code (template_code): Unique identifier
  • Template name (template_name): Template name
  • Template type (template_type): Message type, such as SMS, email, in-app message
  • Template content (content): Message template content, supports variable substitution
  • Status (status): Enabled/Disabled
  • Description (description): Template description

2.1.2 Message Record (MessageRecord)

Attributes:

  • Message type (type): Message type
  • Sender (sender_id): Sender ID
  • Receiver (receiver_id): Receiver ID
  • Message title (title): Message title
  • Message content (content): Message content
  • Send status (send_status): Send status, such as pending, sent, failed
  • Read status (read_status): Read status, such as unread, read
  • Send time (send_time): Send time
  • Read time (read_time): Read time

2.2 Core Functionality

Functions:

  • Message template management
  • Message sending
  • Message receiving
  • Message reading
  • Message statistics

Implementation Details:

2.2.1 Message Sending Mechanism

  • Synchronous Sending: Immediate message sending
  • Asynchronous Sending: Asynchronous sending through message queue, improving system performance
  • Scheduled Sending: Support for scheduled message sending

Message Sending Flow:

  1. Build message object
  2. Select sending method based on message type
  3. Send message
  4. Record message sending status
  5. Notify receiver

2.2.2 Message Receiving Methods

  • In-App Message: In-system message notification
  • Email: Email notification
  • SMS: SMS notification
  • Webhook: Third-party system notification

2.2.3 Message Usage

Send Message:

// Inject message service
@Autowired
private MessageService messageService;

// Send message
MessageDTO messageDTO = new MessageDTO();
messageDTO.setTemplateCode("sys_user_register");
messageDTO.setReceiverId(userId);
messageDTO.setVariables(Collections.singletonMap("username", username));
messageService.send(messageDTO);

Receive Message:

// Get unread message count
import { useMessage } from '@/hooks/useMessage';

const { unreadCount, getUnreadCount } = useMessage();

// Get message list
const messageList = await messageService.getMessageList({ page: 1, size: 10 });

2.3 Best Practices

  • Template Design: Reasonably design message templates, support variable substitution
  • Message Classification: Classify and manage messages based on business requirements
  • Permission Control: Strictly control message sending permissions
  • Performance Optimization: Send messages in batches, avoid frequent calls
  • Message Cleanup: Regularly clean up expired messages to free up storage space

3. Operation Logs

Records critical user operational behaviors for auditing and tracking.

3.1 Data Model

Attributes:

  • Operator (user_id): Operator ID
  • Operator name (username): Operator name
  • Module name (module): Operation module
  • Operation method (method): Operation method
  • Operation type (type): Operation type, such as add, modify, delete
  • Operation IP (ip): Operation IP address
  • Operation time (create_time): Operation time
  • Execution result (status): Execution result, success/failure
  • Error message (error_msg): Error message (if failed)
  • Duration (cost_time): Execution duration (milliseconds)
  • Request parameters (params): Request parameters
  • Return result (result): Return result

3.2 Core Functionality

Functions:

  • Automatic operation log recording
  • Operation log query
  • Operation log export
  • Operation log analysis

Implementation Details:

3.2.1 Log Recording Mechanism

  • Use @OperationLog annotation to automatically capture operation logs
  • Support custom log content
  • Support excluding specific methods

Usage:

@OperationLog(module = "User Management", name = "Add User", type = OperationType.ADD)
@PostMapping
public Result<Void> addUser(@RequestBody @Validated UserDTO userDTO) {
    // Add user logic
}

3.2.2 Log Query

  • Support query by time range
  • Support query by module
  • Support query by user
  • Support query by operation type
  • Support query by execution result

3.3 Best Practices

  • Log Level: Only record critical operations, avoid excessive logs
  • Log Content: Record sufficient context information for problem tracking
  • Performance Optimization: Record logs asynchronously to avoid affecting main business processes
  • Log Retention: Reasonably set log retention period, regularly archive

4. System Config (Config)

Manages dynamic parameters for system runtime, improving system flexibility and configurability.

4.1 Data Model

Attributes:

  • Config key (config_key): Unique identifier
  • Config value (config_value): Config value
  • Config name (config_name): Config name
  • Config type (config_type): Config type, such as string, number, boolean, enum
  • Status (status): Enabled/Disabled
  • Description (description): Config description
  • Sort order (sort): Display order

4.2 Core Functionality

Functions:

  • CRUD operations for config items
  • Config item group management
  • Config item import/export
  • Config item cache management

Implementation Details:

4.2.1 Config Caching Mechanism

  • Use Redis to cache config data
  • Automatically refresh cache when config is updated
  • Support config hot reload

Usage:

// Get config value
ConfigService configService = SpringContextHolder.getBean(ConfigService.class);
String siteTitle = configService.getConfigValue("site_title");

// Or use annotation
@Value("${config.site_title}")
private String siteTitle;

4.3 Best Practices

  • Naming Convention: Config keys use the {module}_{function} format, such as site_title
  • Default Values: Set reasonable default values for config items
  • Type Checking: Strictly validate config value types
  • Permission Control: Strictly control config management permissions

5. File Storage (Storage)

Provides unified file upload and management capabilities, supporting multiple storage modes.

5.1 Supported Modes

  • Local Storage: Files are saved on the server disk
  • Object Storage (OSS): Integration with Alibaba Cloud, Tencent Cloud, MinIO, etc.

5.2 Core Functionality

Functions:

  • File upload
  • File download
  • File preview
  • File deletion
  • File management

Implementation Details:

5.2.1 Local Storage Configuration

# Local storage configuration
storage:
  type: local
  local:
    base-path: uploads
    domain: http://localhost:8080

5.2.2 Object Storage Configuration

# MinIO configuration
storage:
  type: minio
  minio:
    endpoint: http://minio:9000
    access-key: minioadmin
    secret-key: minioadmin
    bucket-name: tagtag
    domain: http://minio:9000

5.2.3 File Usage

Upload File:

// Inject storage service
@Autowired
private StorageService storageService;

// Upload file
MultipartFile file = request.getFile("file");
StorageResult result = storageService.upload(file, "image/");

Download File:

// Download file
String fileId = request.getParameter("fileId");
storageService.download(fileId, response);

5.3 Best Practices

  • File Naming: Use unique identifiers to name files, avoid filename conflicts
  • File Type: Restrict allowed file types for upload
  • File Size: Set reasonable file size limits
  • Security Measures: Scan uploaded files for viruses
  • Backup Strategy: Regularly backup important files

9. Best Practices

9.1 System Design

  • Modular Design: Each functional module is independent, facilitating extension and maintenance
  • Configurability: Core parameters support dynamic configuration
  • High Performance: Use caching to improve system performance
  • High Availability: Critical functions support fault tolerance

9.2 Development Standards

  • Naming Convention: Unified naming convention to improve code readability
  • Comment Convention: Comprehensive code comments for easier maintenance
  • Testing Convention: Write unit tests for core functions
  • Documentation Convention: Comprehensive interface documentation and usage documentation

9.3 Deployment Recommendations

  • Environment Isolation: Separate development, testing, and production environments
  • Configuration Isolation: Use different configuration files for different environments
  • Monitoring and Alerting: Configure comprehensive monitoring and alerting mechanisms
  • Backup Strategy: Regularly backup data and configuration

11. Summary

The System module is one of the core modules of the Tagtag Starter system, responsible for managing global configuration, dictionary data, message management, log monitoring, and file storage services.

Through the System module, administrators can conveniently manage system configuration, maintain dictionary data, send message notifications, view operation logs, and manage file storage. At the same time, the System module provides comprehensive API interfaces for other modules to call.

The design and implementation of the System module follow the principles of modularity and extensibility, facilitating future feature expansion and customized development.
