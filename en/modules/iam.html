IAM (Identity & Access)

Documentation for Identity and Access Management module.

The IAM (Identity and Access Management) module is responsible for user authentication, authorization, and organizational structure management.

1. RBAC Permission Model

The IAM module is designed based on the RBAC (Role-Based Access Control) model, implementing fine-grained permission control.

1.1 RBAC Model Overview

RBAC is a role-based access control model that achieves centralized management and flexible allocation of permissions by assigning permissions to roles and then assigning roles to users.

Core Concepts:

  • Many-to-many relationship between Users and Roles
  • Many-to-many relationship between Roles and Permissions
  • One-to-one relationship between Permissions and Resources

1.2 Permission Types

Tagtag Starter's RBAC model supports two types of permissions:

1.2.1 Functional Permissions

Control whether a user can access a specific feature or operation in the system, including:

  • Menu access permissions
  • Button operation permissions
  • API interface permissions

Functional permission identifiers follow the format: {module}:{resource}:{operation}, for example:

  • sys:user:view - View users
  • sys:user:add - Add user
  • sys:user:update - Update user
  • sys:user:delete - Delete user

1.2.2 Data Permissions

Control the data range that users can access, including:

  • All data: Can access all data
  • Department data: Can only access data belonging to their own department
  • Department and sub-department data: Can access data belonging to their department and its sub-departments
  • Custom data: Filter data based on custom rules

1.3 Permission Inheritance

Permissions support inheritance relationships:

  • Directory permission inheritance: If a user has access to a directory, they automatically have access to all submenus under that directory
  • Role permission inheritance: You can create role hierarchies where child roles inherit permissions from parent roles

2. Data Model

2.1 User

Represents a system operator.

Attributes:

  • Username (username)
  • Password (password, BCrypt encrypted)
  • Name (name)
  • Email (email)
  • Phone (phone)
  • Status (status: 1-enabled, 0-disabled)
  • Department (dept_id)
  • Create time (create_time)
  • Update time (update_time)

Associations:

  • A user can have multiple roles (many-to-many relationship)
  • A user belongs to one department (many-to-one relationship)

2.2 Role

Represents a collection of permissions.

Attributes:

  • Role name (name)
  • Role code (code)
  • Status (status: 1-enabled, 0-disabled)
  • Data permission type (data_scope_type)
  • Description (description)

Functions:

  • Used to batch assign permissions to users
  • Can configure the data access range for the role
  • Supports role hierarchies to implement permission inheritance

2.3 Permission/Menu

Represents functional resources in the system.

Attributes:

  • Permission name (name)
  • Permission code (code)
  • Permission type (type: 1-directory, 2-menu, 3-button)
  • Parent ID (parent_id)
  • Route path (path)
  • Component path (component)
  • Icon (icon)
  • Sort order (sort)
  • Status (status: 1-enabled, 0-disabled)

Types:

  • Directory: Grouping in the sidebar menu, does not correspond to a specific page
  • Menu: Specific page entry, corresponds to a route
  • Button: Functional operations within a page, corresponds to a permission identifier

Identifier: e.g., sys:user:add, used for backend API permission checks and frontend button display control.

2.4 Department (Dept)

Represents the organizational hierarchy of the enterprise.

Attributes:

  • Department name (name)
  • Department code (code)
  • Parent ID (parent_id)
  • Sort order (sort)
  • Status (status: 1-enabled, 0-disabled)

Structure: Tree structure, supporting infinite levels.

Usage: Used to build the organizational tree and as a basis for data permission judgment.

2.5 Relationship Tables

  • user_role: Many-to-many relationship table between users and roles
  • role_permission: Many-to-many relationship table between roles and permissions

3. Core Functionality Implementation

3.1 User Management

Functions: User CRUD operations, password reset, status change, role assignment.

Implementation Details:

  • Password encryption: Uses BCrypt algorithm for password encryption to ensure password security
  • User status control: Disabled users cannot log in to the system
  • Role assignment: Supports assigning multiple roles to a user
  • Data permissions: Controls the data range users can access based on their department and role data permission configuration

3.2 Role Management

Functions: Role creation and maintenance, permission assignment, data permission configuration.

Implementation Details:

  • Permission assignment: Check menu and button permissions through a tree selector
  • Data permission configuration: Supports configuring the data access range for roles
  • Role status control: Permissions of disabled roles are invalid for users

3.3 Menu Management

Functions: Visual management of the system menu structure, configure menu attributes, define button permissions.

Implementation Details:

  • Tree structure: Supports infinite-level menus
  • Route configuration: Automatically generates frontend route configuration
  • Component path: Configure page component path
  • Permission identifier: Define button permission identifiers for frontend button control

3.4 Department Management

Functions: Maintain the enterprise organizational tree.

Implementation Details:

  • Tree structure: Supports infinite-level departments
  • Department code: Automatically generates department codes, supports custom code rules
  • Department status: Users under disabled departments cannot log in to the system

4. Permission Verification Flow

4.1 Backend Permission Verification

Flow:

  1. User logs in to the system and obtains a JWT token
  2. When requesting an API, carry the JWT token in the request header
  3. Backend interceptor verifies the validity of the JWT token
  4. Permission guard @PreAuthorize checks if the user has access permission for that API
  5. If the user does not have permission, return a 403 Forbidden response

Implementation:

@RestController
@RequestMapping("/users")
public class UserController {
    
    @PostMapping
    @PreAuthorize("@ss.hasPermission('sys:user:add')")
    public Result<Void> addUser(@RequestBody @Validated UserDTO userDTO) {
        // Add user logic
    }
    
    @PutMapping
    @PreAuthorize("@ss.hasPermission('sys:user:update')")
    public Result<Void> updateUser(@RequestBody @Validated UserDTO userDTO) {
        // Update user logic
    }
}

Permission Guard Implementation:

@Component("ss")
public class PermissionGuard {
    
    public boolean hasPermission(String permission) {
        // 1. Get current user
        UserPrincipal user = AuthContext.getCurrentUser();
        
        // 2. Check if user is super admin
        if (user.isSuperAdmin()) {
            return true;
        }
        
        // 3. Check if user has this permission
        return user.getPermissions().contains(permission);
    }
}

4.2 Frontend Permission Control

Flow:

  1. After successful login, get the user's menu list and permission list
  2. Dynamically generate the left sidebar menu based on the menu list
  3. Control page button display/hide based on the permission list

Implementation:

4.2.1 Menu Permission Control

// Get user menus
const menuList = await getMenuList();
// Dynamically generate routes
const routes = generateRoutes(menuList);
// Add to route table
routes.forEach(route => router.addRoute(route));

4.2.2 Button Permission Control

Use the v-auth directive to control button display:

<template>
  <button v-auth="'sys:user:add'">Add User</button>
  <button v-auth="'sys:user:delete'">Delete User</button>
</template>

Or use the AccessControl component:

<template>
  <AccessControl :access="'sys:user:add'">
    <button>Add User</button>
  </AccessControl>
</template>

Permission Directive Implementation:

export const AuthDirective = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const permission = binding.value;
    // Check if user has this permission
    if (!useAccess().hasPermission(permission)) {
      el.style.display = 'none';
    }
  }
};

8. Best Practices

8.1 Permission Design

  • Principle of Least Privilege: Only grant users the minimum permissions needed to complete their work
  • Permission Grouping: Group permissions by functional modules for easier management and assignment
  • Naming Convention: Use a unified naming convention for permission identifiers, such as {module}:{resource}:{operation}
  • Data Permissions: Configure role data permissions reasonably based on business requirements

8.2 Role Design

  • Role Layering: Create different levels of roles, such as super administrator, administrator, and regular user
  • Separation of Duties: Different roles are responsible for different functional modules
  • Role Reuse: Avoid creating dedicated roles for each user; try to reuse existing roles

8.3 Menu Design

  • Reasonable Classification: Classify menus by functional modules for easy user navigation
  • Appropriate Depth: Menu levels should not be too deep; it is recommended not to exceed 3 levels
  • Permission Association: Each menu and button should be associated with corresponding permission identifiers
  • Default Expansion: Reasonably set default expanded menus to improve user experience

8.4 Department Design

  • Clear Hierarchy: Department levels should clearly reflect the enterprise's organizational structure
  • Coding Convention: Use a unified department coding rule
  • Status Management: Update department status in a timely manner and disable unnecessary departments

9. Extension Suggestions

9.1 Support Multi-Tenancy

If the system needs to support multi-tenancy, you can add tenant management functionality to the IAM module:

  • Add tenant entity
  • Add tenant attributes to users and departments
  • Implement tenant isolation

9.2 Support Third-Party Login

Integrate third-party login, such as OAuth 2.0, LDAP, etc.:

  • Add third-party login configuration
  • Implement third-party login callback handling
  • Associate third-party users with system users

9.3 Support Fine-Grained Data Permissions

Implement more fine-grained data permission control:

  • Support field-based data permissions
  • Support condition-based data permissions
  • Support custom data permission rules

9.4 Support Permission Auditing

Add permission auditing functionality to record user permission change history:

  • Record user role changes
  • Record role permission changes
  • Generate permission audit reports

10. Summary

The IAM module is one of the core modules of the Tagtag Starter system, responsible for user authentication, authorization, and organizational structure management. Designed based on the RBAC model, it implements fine-grained permission control, including functional permissions and data permissions.

Through the IAM module, administrators can conveniently manage users, roles, menus, and departments, achieving centralized management and flexible allocation of permissions. At the same time, the system provides a comprehensive permission verification mechanism to ensure that only authorized users can access system resources.

The design and implementation of the IAM module follow the principles of modularity and extensibility, facilitating future feature expansion and customized development.
