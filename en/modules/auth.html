Authentication Module

Authentication and authorization module documentation, detailing JWT authentication flow with a professional yet witty touch.

The Auth module is the security gatekeeper of the system, responsible for verifying user identities and granting access permissions. Tagtag's authentication module adopts a modern JWT (JSON Web Token) authentication mechanism, implementing stateless authentication that improves system scalability and performance—because who wants to manage sessions when you can just sign tokens and be done with it?

1. Authentication Module Overview

The authentication module is the security core of the system, acting as the bouncer at the club entrance—checking IDs, verifying credentials, and deciding who gets in. Tagtag Starter's authentication module uses JWT for stateless authentication, which means no more session headaches and better scalability. Win-win!

1.1 Core Features

  • User login and logout (the classic "knock and enter" routine)
  • JWT token generation and verification (digital signatures that actually mean something)
  • Token refresh mechanism (because tokens expire, unlike our love for good coffee)
  • Permission verification (making sure users only see what they're supposed to)
  • Session management (or rather, the lack thereof—thanks to JWT!)

1.2 Technology Stack

  • JWT: JSON Web Token, used for stateless authentication. It's like a digital passport that carries all your info.
  • BCrypt: Password encryption algorithm to ensure password security. Because storing plain text passwords is so 1990s.
  • Spring Security: Security framework providing authentication and authorization features. The heavy lifter in our security stack.
  • Redis: Used for rate limiting and temporary storage. Fast, reliable, and always there when you need it.

2. JWT Authentication Principles

2.1 JWT Basic Concepts

JWT (JSON Web Token) is an open standard (RFC 7519) used to securely transmit information between parties as a JSON object. Think of it as a digital envelope that contains your credentials, signed by the server so nobody can tamper with it. It's like sending a letter with a wax seal—if the seal is broken, you know something's fishy.

2.2 JWT Structure

JWT consists of three parts separated by dots (.):

  • Header: Contains the token type and signature algorithm. The "what and how" of the token.
  • Payload: Contains claims, such as user ID, roles, expiration time, etc. The "who, what, and when" of the user.
  • Signature: Used to verify the integrity of the token. The "proof it's legit" part.

Format: header.payload.signature

It's like a sandwich: the header is the top bun, the payload is the tasty filling, and the signature is the bottom bun that holds it all together.

2.3 JWT Workflow

  1. User logs in to the system using username and password (and captcha, because bots are annoying)
  2. Server verifies the correctness of the username and password (and checks the captcha)
  3. Server generates a JWT token and returns it to the client (access token for short-term use, refresh token for the long game)
  4. Client stores the JWT token (usually in localStorage or memory)
  5. Client carries the JWT token in subsequent requests (in the Authorization header, naturally)
  6. Server verifies the validity of the JWT token (signature, expiration, and version check)
  7. Server processes the request and returns a response (if everything checks out)

2.4 JWT Advantages

  • Stateless: Server doesn't need to store session information, improving system scalability. Less memory usage, happier servers.
  • Cross-domain Support: JWT supports cross-domain authentication, facilitating distributed system construction. One token to rule them all.
  • Self-contained: Token contains all necessary user information, reducing database queries. Everything you need is right there in the token.
  • Easy Integration: Compatible with various clients. Web, mobile, IoT—if it can make HTTP requests, it can use JWT.

4. Core Function Implementation

4.1 User Login

Function: Verify user identity and generate JWT tokens.

Implementation Details:

  • Captcha verification: Prevents automated attacks (because bots don't solve captchas)
  • Password verification: Uses BCrypt algorithm to verify password (slow and secure, just how we like it)
  • Token generation: Generates access token and refresh token (two tokens are better than one)
  • Token version: Includes token version in claims for logout functionality
  • Rate limiting: Prevents brute force attacks (10 attempts per minute per username+IP)
  • Response format: Returns access token and refresh token

Login Request:

POST /auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123",
  "captcha": {
    "code": "ABCD",
    "captchaId": "uuid-here"
  }
}

Login Response:

{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "tokenType": "Bearer",
    "expiresIn": 3600
  }
}

Note: The response includes both access token (short-lived, typically 1 hour) and refresh token (long-lived, typically 7 days). The access token is used for API requests, while the refresh token is used to get a new access token when it expires.

4.2 Token Verification

Function: Verify the validity of JWT tokens.

Implementation Details:

  • Signature verification: Verifies if the token signature is valid (no tampering allowed!)
  • Expiration verification: Checks if the token has expired (time waits for no one, not even tokens)
  • Token version verification: Checks if the token version matches the current version in database (logout mechanism)
  • Permission verification: Verifies if the user has access permissions (because not everyone should see everything)

Implementation Code:

public boolean validateToken(String token) {
    try {
        SignedJWT jwt = SignedJWT.parse(token);

        if (!signerStrategy.verify(jwt)) {
            return false;
        }

        Date exp = jwt.getJWTClaimsSet().getExpirationTime();
        return exp != null && Instant.now().isBefore(exp.toInstant());
    } catch (Exception e) {
        log.debug("Token validation failed", e);
        return false;
    }
}

4.3 Token Refresh

Function: Refresh access token using refresh token.

Implementation Details:

  • Refresh token validation: Verifies the refresh token is valid
  • Token version check: Ensures the refresh token hasn't been invalidated by logout
  • New token generation: Generates new access and refresh tokens
  • Rate limiting: Prevents abuse (30 refresh attempts per minute per IP)

Refresh Request:

POST /auth/refresh
Content-Type: application/json

{
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

Refresh Response:

{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "tokenType": "Bearer",
    "expiresIn": 3600
  }
}

Pro Tip: Always use the refresh token to get a new access token before it expires. Don't wait until the last second—it's like renewing your passport before an international trip.

4.4 User Logout

Function: Log out and invalidate the token.

Implementation Details:

  • Token validation: Validates the access token before processing logout
  • Token version bump: Increments the user's token version, invalidating all existing tokens
  • No blacklist needed: Because we use token versioning instead of blacklisting (cleaner and more efficient)
  • Rate limiting: Not needed for logout (let people leave whenever they want)

Logout Request:

POST /auth/logout
Content-Type: application/json

{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

Logout Response:

{
  "success": true,
  "message": "Logout successful"
}

How it works: When a user logs out, we increment their token version in the database. All existing tokens (both access and refresh) contain the old version number, so they'll fail validation on subsequent requests. It's like changing the locks when someone moves out—old keys just don't work anymore.

4.5 User Registration

Function: Register a new user account.

Implementation Details:

  • Captcha verification: Prevents automated registration (bots can't solve captchas, hopefully)
  • Username uniqueness check: Ensures username is not already taken (no duplicates allowed)
  • Password strength validation: Enforces strong password requirements (at least 8 characters, uppercase, lowercase, number, and special character)
  • Password encryption: Uses BCrypt to encrypt the password before storing (security first!)
  • Rate limiting: Prevents abuse (10 registration attempts per minute per IP)

Registration Request:

POST /auth/register
Content-Type: application/json

{
  "username": "newuser",
  "password": "SecurePass123!",
  "captcha": {
    "code": "ABCD",
    "captchaId": "uuid-here"
  }
}

Registration Response:

{
  "success": true,
  "message": "Registration successful"
}

Password Requirements: Your password must be at least 8 characters long and include:

  • At least one uppercase letter (A-Z)
  • At least one lowercase letter (a-z)
  • At least one digit (0-9)
  • At least one special character (!@#$%^&* etc.)

Because "password123" just doesn't cut it anymore.

4.6 Get Current User Info

Function: Get the current logged-in user's information.

Implementation Details:

  • Authentication required: Must provide valid access token
  • Password excluded: Password field is set to null in response (security best practice)
  • User details: Returns user ID, username, name, roles, and other profile information

Request:

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Response:

    "name": "Administrator",
    "email": "admin@tagtag.dev",
    "status": 1,
    "roleIds": [1, 2],
    "password": null
  }
}

4.7 Get User Permissions

Function: Get the current user's permission codes.

Implementation Details:

  • Authentication required: Must provide valid access token
  • Admin check: If user is super admin, returns all permission codes
  • Role-based: For regular users, resolves permissions based on assigned roles
  • Permission codes: Returns set of permission strings (e.g., "sys:user:view", "sys:user:add")

Request:

GET /auth/codes
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Response:

{
  "success": true,
  "data": [
    "sys:user:view",
    "sys:user:add",
    "sys:user:edit",
    "sys:user:delete",
    "sys:role:view",
    "sys:menu:view"
  ]
}

4.8 Get User Routes

Function: Get the current user's accessible routes (menus).

Implementation Details:

  • Authentication required: Must provide valid access token
  • Admin check: If user is super admin, returns all menu routes
  • Role-based filtering: For regular users, filters menus based on assigned roles
  • Route conversion: Converts menu tree to route records for frontend routing
  • Button exclusion: Button-type menus are not included in routes (they're for permissions, not navigation)

6. Frontend Integration

6.1 Login Page

The login page collects username, password, and captcha, then calls the login API:

import { loginApi } from '#/api/core/auth';

async function handleLogin() {
  const { username, password, captcha } = formData;
  const result = await loginApi({
    username,
    password,
    captcha: {
      code: captcha.code,
      captchaId: captcha.id
    }
  });

  if (result.accessToken) {
    // Store tokens
    accessStore.setAccessToken(result.accessToken);
    accessStore.setRefreshToken(result.refreshToken);

    // Fetch user info and permissions
    const [userInfo, accessCodes] = await Promise.all([
      fetchUserInfo(),
      getAccessCodesApi()
    ]);

    userStore.setUserInfo(userInfo);
    accessStore.setAccessCodes(accessCodes);

    // Redirect to home page
    router.push(userInfo.homePath || '/dashboard');
  }
}

6.2 Token Management

Tokens are managed in the auth store:

import { useAuthStore } from '#/store/auth';

const authStore = useAuthStore();

// Login
await authStore.authLogin({
  username: 'admin',
  password: 'admin123',
  captcha: {
    code: 'ABCD',
    captchaId: 'uuid'
  }
});

// Logout
await authStore.logout();

// Get user info
const userInfo = await authStore.fetchUserInfo();

6.3 Request Interceptor

The request interceptor automatically adds JWT tokens to requests:

requestClient.interceptors.request.use((config) => {
  const token = accessStore.accessToken;
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

6.4 Response Interceptor

The response interceptor handles token expiration and refresh:

requestClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      // Token expired, try to refresh
      const refreshToken = accessStore.refreshToken;
      if (refreshToken) {
        try {
          const result = await refreshTokenApi({ refreshToken });
          accessStore.setAccessToken(result.accessToken);
          accessStore.setRefreshToken(result.refreshToken);

          // Retry the original request
          error.config.headers.Authorization = `Bearer ${result.accessToken}`;
          return requestClient(error.config);
        } catch (refreshError) {
          // Refresh failed, redirect to login
          await authStore.logout();
        }
      } else {
        // No refresh token, redirect to login
        await authStore.logout();
      }
    }
    return Promise.reject(error);
  }
);

7. Security Best Practices

7.1 Password Security

  • Use strong passwords: At least 8 characters with uppercase, lowercase, numbers, and special characters
  • Never store plain text passwords: Always use BCrypt or similar hashing algorithms
  • Implement rate limiting: Prevent brute force attacks
  • Use HTTPS: Always transmit passwords over secure connections

7.2 Token Security

  • Use short-lived access tokens: Typically 1 hour or less
  • Use long-lived refresh tokens: Typically 7 days or less
  • Store tokens securely: Use httpOnly cookies or secure storage mechanisms
  • Implement token versioning: Allow for immediate token invalidation on logout
  • Validate tokens on every request: Check signature, expiration, and version

7.3 Rate Limiting

  • Login rate limit: 10 attempts per minute per username+IP
  • Refresh rate limit: 30 attempts per minute per IP
  • Registration rate limit: 10 attempts per minute per IP
  • Use Redis for distributed rate limiting

7.4 Captcha

  • Use captchas on login and registration: Prevent automated attacks
  • Implement captcha validation: Always validate captcha before processing
  • Use secure captcha generation: Prevent captcha prediction

8. Common Issues and Solutions

8.1 Token Expired

Problem: API requests return 401 Unauthorized error.

Solution: The frontend should automatically refresh the token using the refresh token. If refresh fails, redirect to login page.

8.2 Invalid Token

Problem: Token validation fails even though it hasn't expired.

Solution: Check if the token version matches the current version in the database. This usually happens after logout—just log in again.

8.3 Captcha Validation Failed

Problem: Captcha validation fails even though the code is correct.

Solution: Captchas are single-use. Make sure you're using the latest captcha ID and code, and that the captcha hasn't expired.

8.4 Password Too Weak

Problem: Registration fails with "password too weak" error.

Solution: Ensure your password meets the strength requirements: at least 8 characters with uppercase, lowercase, numbers, and special characters.

9. Extension Suggestions

9.1 Support Multiple Login Methods

  • SMS Login: Support SMS verification code login (because passwords are so 2010)
  • Email Login: Support email verification code login
  • Third-party Login: Support WeChat, QQ, GitHub and other third-party logins (let users bring their own identities)
  • Biometric Login: Support fingerprint, facial recognition and other biometric logins (because your face is your password)

9.2 Enhance Security Mechanisms

  • Multi-factor Authentication: Support multi-factor authentication to improve security (because one factor is never enough)
  • Risk Control: Implement login risk control to prevent brute force attacks
  • Audit Logs: Record authentication-related audit logs (keep an eye on who's doing what)
  • Anomaly Detection: Detect abnormal login behaviors (like logging in from Antarctica when you're in New York)

9.3 Support SSO

Implement Single Sign-On (SSO) to support seamless login between multiple systems. Log in once, access everywhere—it's like having a VIP pass to all the clubs.

9.4 Support JWT Extensions

  • Support JWT Extension Claims: Support custom claims for additional user information
  • Support JWT Encryption: Support JWT encryption to improve security (signatures are good, encryption is better)
  • Support JWT Revocation: Support more granular JWT revocation mechanisms

10. Summary

The authentication module is the security core of the Tagtag system, implementing stateless authentication mechanism based on JWT. This module provides complete authentication functions, including user login, token verification, token refresh, user logout, user registration, and permission management.

Key features:

  • JWT-based stateless authentication for scalability
  • Token versioning for secure logout
  • BCrypt password encryption for security
  • Rate limiting for abuse prevention
  • Captcha verification for bot protection
  • Comprehensive API endpoints for all authentication needs

By following the authentication module's design and implementation, you can build a secure, scalable, and user-friendly authentication system for your application. And remember: security is not a product, it's a process. Stay vigilant, keep learning, and never stop improving.
