Architecture Overview

High-level overview of the Tagtag architecture.

Tagtag adopts a modern architecture with separation of frontend and backend. The frontend handles user interaction and interface display, while the backend is responsible for business logic and data processing. The system is designed with modularity, high cohesion, and low coupling principles for easy extension and maintenance. Simply put, it's about clarifying responsibilities - each part does its job without interfering with others.

The Tagtag system consists of two main parts:

  • Frontend: Located in the frontend directory. A modern Single Page Application (SPA) built with Vue 3 + Vite + TypeScript, adopting the Vben Admin 5 framework, and using Tailwind CSS and Shadcn UI for interface development. Beautiful interface, smooth interaction, and top-notch user experience.
  • Backend: Multiple Java modules located in the backend directory. A modular application built with Spring Boot, responsible for providing RESTful APIs and data persistence services. Stable, reliable, and excellent performance - like a hardworking ox that never complains.

Interaction

  • Protocol: Frontend and backend communicate via HTTP/HTTPS protocol using RESTful API style. Simple, standard, and easy to understand.
  • Data Format: JSON is used as the standard data exchange format. Lightweight, readable, and easy to parse.
  • Authentication: Stateless authentication using JWT (JSON Web Token). Secure, efficient, and stateless.

Frontend-Backend Interaction Flow

  1. User Authentication Flow:
    • User accesses the frontend application, enters username and password
    • Frontend calls the login API, sends authentication request
    • Backend verifies username and password, generates JWT token
    • Frontend stores JWT token, carries token in subsequent requests

    This process is like withdrawing money from a bank - first verify your identity, get a voucher, and then you can handle business with that voucher.
  2. API Request Flow:
    • Frontend initiates API request, carries JWT token
    • Backend filter verifies token validity
    • Controller receives request, calls service layer to process business logic
    • Service layer calls data layer to access database
    • Data layer returns results, service layer encapsulates business response
    • Controller returns JSON response to frontend
    • Frontend processes response, updates interface

    This flow is like ordering food delivery - you place an order (frontend request), the merchant accepts it (controller), the chef cooks (service layer), packaging (data layer), delivery (return response), and finally you receive your food (frontend processing).
  3. WebSocket Real-time Communication (optional):
    • Frontend establishes WebSocket connection, carries authentication information
    • Backend verifies and establishes connection
    • Both parties exchange real-time data via WebSocket

    This is like making a phone call - once the connection is established, both parties can talk anytime without redialing.

Backend Architecture

Tagtag backend follows a strict multi-module architecture to separate infrastructure, business logic, and API definitions. This design makes the system's components have clear responsibilities, facilitating maintenance and extension. Simply put, it's about clarifying responsibilities - each part does its job without interfering with others.

Backend Layered Design

The backend project is organized into the following layers:

1. Common Layer (tagtag-common)

Pure utility classes with no Spring dependencies, can be used independently of the Spring framework.

  • Constants
  • Exceptions
  • Base Models (PageResult, Result)
  • Utils (BeanUtils, StringUtils, DateUtils)
    This layer is like a toolbox containing various tools that anyone can use without worrying about dependencies.

2. Framework Layer (tagtag-framework)

Infrastructure integration. This layer depends on Spring Boot and other third-party libraries, providing technical support for upper layers.

  • MyBatis Plus Configuration
  • Redis Configuration
  • Spring Security & JWT
    • JWT Service and Strategy
    • Permission Guards
    • Authentication Context Management
  • Global Exception Handling
  • Aspect definitions (RateLimit)
  • MapStruct Configuration
  • Web Configuration (CORS, TraceId)
    This layer is like the foundation, providing stable technical support for upper layers. Without this layer, the upper layers cannot be built.

3. Kernel Layer (tagtag-kernel)

Core business support logic. Provides common support and extension points needed for business development.

  • Annotations (@RateLimit, @RequirePerm, @RequireRole)
  • Enums (Status, Gender)
  • Business constants and error codes
  • Security-related constants and claims
    This layer is like core business rules, defining the basic norms and constraints for business development.

4. Contract Layer (tagtag-contract)

Defines interfaces and data structures between modules, adopting contract-first design.

  • DTOs: Data Transfer Objects used for API input/output.
  • API Interfaces: Java interfaces defining the service contracts.
  • VO: View Objects, used for data encapsulation for frontend display.
  • Enums: Business-related enum definitions.
    This layer is like a contract, defining the interfaces and data structures between parties (modules). With a contract, everyone knows how to cooperate.

5. Module Layer (tagtag-module)

The actual business implementation, each module corresponds to a specific business domain.

  • Controllers: Handle HTTP requests, call service layer.
  • Service Implementations: Implement business logic.
  • Entity classes: Map to database tables.
  • Mappers: Database operation interfaces.
  • Business Models: Domain models and business rules.
    This layer is like various departments, each department responsible for its own business, each doing its own job.

6. Start Layer (tagtag-start)

Application entry point. Aggregates all modules and provides configuration files.

  • Spring Boot main application class
  • Configuration files (application.yml)
  • Environment variable configuration
  • Dependency management
    This layer is like the company gate, all modules enter and exit from here.

Backend Module Design Principles

  1. High cohesion, low coupling: Each module is responsible for only one business domain, and modules communicate through the contract layer. Like various departments, each doing its own job, cooperating through contracts (contract layer).
  2. Dependency inversion: High-level modules depend on abstractions, not on concrete implementations. Like a boss depends on contracts, not on specific employees.
  3. Open-closed principle: Open for extension, closed for modification. Like software upgrades, you can add new features but cannot change old functionality.
  4. Single responsibility: Each class and method is responsible for only one function. Like a person only responsible for one thing, won't do everything.
  5. Layered design: Strictly follow layered architecture, no cross-layer calls. Like company hierarchy, cannot skip levels to report.

Module Division

The frontend project is mainly divided into three layers:

1. Application Layer (apps)

Contains specific application implementations, each subdirectory corresponds to an independent frontend application.

  • apps/tagtag: Main application, containing complete business logic and interface.
    • src/api: API request definitions and encapsulation.
    • src/views: Page view components.
    • src/router: Routing configuration and guards.
    • src/store: Pinia state management.
    • src/layouts: Layout components.
    • src/components: Business components.

2. Core Package Layer (packages)

Provides reusable core functions and components for the application layer.

  • packages/@core: Core shared library.
    • base: Basic design system, icons and shared tools.
    • ui-kit: Encapsulated UI component library.
      • shadcn-ui: Basic components based on Shadcn UI.
      • form-ui: Form component encapsulation.
      • layout-ui: Layout component encapsulation.
      • popup-ui: Popup, drawer and other component encapsulation.
    • effects: Business side effects and logic encapsulation.
      • access: Permission control logic.
      • hooks: Common Hooks (such as useTabs, useRefresh).
      • request: Request library encapsulation.
      • plugins: Third-party plugin integration (Echarts, Motion, Vxe-table).
    • preferences: Application preference settings management.

3. Internal Tools Layer (internal)

Provides tools and configuration for building, development and maintenance.

  • internal/lint-configs: ESLint, Prettier, Stylelint and other code style configuration.
  • internal/vite-config: Vite build configuration encapsulation.
  • internal/tailwind-config: Tailwind CSS configuration encapsulation.
  • internal/tsconfig: TypeScript configuration files.

Frontend Design Principles

  1. Component-based design: Split UI into reusable components to improve code reusability.
  2. Separation of concerns: Separation of business logic, UI display and state management.
  3. Type safety: Fully utilize TypeScript's type checking.
  4. Responsive design: Adapt to different screen sizes.
  5. Performance optimization: Code splitting, lazy loading, virtual scrolling and other optimization methods.
  6. Accessibility: Ensure the application is accessible to all users, including people with disabilities.
  7. Internationalization support: Support multi-language switching.

Monorepo Advantages

  1. Code sharing: Multiple applications can share components, utility classes and configurations.
  2. Unified dependency management: All applications use the same version of dependencies, avoiding version conflicts.
  3. Unified build tools: Shared build configurations and scripts, improving development efficiency.
  4. Better collaboration: Team members can share and reuse code more conveniently.
  5. Simplified version management: Unified version number management, simplifying release process.
